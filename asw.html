SWC Development

An AUTOSAR Software Component (SWC) is the fundamental building block of the application layer. Each SWC encapsulates a well-defined piece of functionality and communicates exclusively through ports defined in ARXML.

SWC Types

TypeDescriptionUse Case
ApplicationSwComponentTypeStandard application componentControl logic, state machines
SensorActuatorSwComponentTypeInterfaces with hardware I/OSensor read, actuator drive
ComplexDeviceDriverSwComponentTypeDirect hardware access (bypasses MCAL APIs)Custom peripherals
ServiceSwComponentTypeProvides BSW services to ASWDiagnostics proxy, NvM proxy
ParameterSwComponentTypeCalibration parameter holderTunable parameter sets

SWC Internal Structure

Each SWC contains:

  • Ports — Provide-ports (PPort) and Require-ports (RPort) defining the component interface
  • Internal Behavior — Runnables, events, exclusive areas, per-instance memory
  • Implementation — The actual C code implementing the runnables

Port Interface Types

  • Sender-Receiver (S/R) — Asynchronous data exchange with implicit or explicit access
  • Client-Server (C/S) — Synchronous or asynchronous function call pattern
  • Mode-Switch — Mode notifications (e.g., EcuM mode changes)
  • Trigger — Event-driven trigger signals
  • NvData — Non-volatile data interface for NvM access
Best Practice: Keep SWCs cohesive and single-purpose. One SWC should represent one logical function. Avoid monolithic SWCs that handle multiple unrelated responsibilities.

CDD Development

A Complex Device Driver (CDD) provides a standardized SWC interface while having direct access to microcontroller hardware registers without going through the MCAL abstraction layer.

When to Use a CDD

  • Hardware peripherals not covered by standard MCAL drivers
  • Time-critical operations requiring direct register access
  • Custom communication protocols not handled by standard stacks
  • Legacy driver integration into an AUTOSAR architecture

Warning: CDDs bypass the AUTOSAR abstraction layers. They reduce portability and increase the effort needed during ECU migration. Use only when standard MCAL drivers are insufficient.

Runnable Design

A Runnable Entity is the smallest schedulable unit of code in an AUTOSAR SWC. Each runnable is triggered by an RTE Event and mapped to an OS Task.

RTE Event Types

Event TypeTriggerTypical Use
TimingEventPeriodic timer (e.g., 10ms)Cyclic control loop
DataReceivedEventNew data on an RPortEvent-driven processing
InitEventSWC initializationVariable init, resource setup
ModeSwitchEventMode entry/exitState-dependent behavior
OperationInvokedEventC/S operation callServer-side handler
DataWriteCompleteEventExplicit write completeConfirmation handling

Runnable-to-Task Mapping

Each runnable must be mapped to an OS task. The mapping strategy affects real-time behavior:

  • One runnable per task — Maximum isolation, highest overhead
  • Multiple runnables per task — Shared task context, sequential execution within task
  • Category 1 vs Category 2 runnables — Cat1 runnables can preempt Cat2 and vice versa depending on task priority
Tip: Group runnables with the same period into the same task to minimize context switch overhead while maintaining deterministic scheduling.

ARXML Explained

ARXML (AUTOSAR XML) is the standardized exchange format for all AUTOSAR configuration and design data. Every element in an AUTOSAR system — from data types to SWC descriptions to BSW module configurations — is described in ARXML files.

Key ARXML Elements

ElementPurpose
AR-PACKAGENamespace container for organizing elements
APPLICATION-SW-COMPONENT-TYPESWC type definition with ports
SENDER-RECEIVER-INTERFACES/R port interface definition
CLIENT-SERVER-INTERFACEC/S port interface definition
SWC-INTERNAL-BEHAVIORRunnables, events, exclusive areas
SYSTEM-MAPPINGSWC-to-ECU allocation
IMPLEMENTATION-DATA-TYPEC-level data type mapping
COMPOSITION-SW-COMPONENT-TYPEHierarchical SWC aggregation

Example: SWC Port Definition

<APPLICATION-SW-COMPONENT-TYPE>
  <SHORT-NAME>MyControlSWC</SHORT-NAME>
  <PORTS>
    <R-PORT-PROTOTYPE>
      <SHORT-NAME>SensorInput</SHORT-NAME>
      <REQUIRED-INTERFACE-TREF DEST="SENDER-RECEIVER-INTERFACE">
        /Interfaces/SensorDataIf
      </REQUIRED-INTERFACE-TREF>
    </R-PORT-PROTOTYPE>
    <P-PORT-PROTOTYPE>
      <SHORT-NAME>ActuatorOutput</SHORT-NAME>
      <PROVIDED-INTERFACE-TREF DEST="SENDER-RECEIVER-INTERFACE">
        /Interfaces/ActuatorCmdIf
      </PROVIDED-INTERFACE-TREF>
    </P-PORT-PROTOTYPE>
  </PORTS>
</APPLICATION-SW-COMPONENT-TYPE>

XDM Modeling

The XDM (XML Data Model) approach defines the structure and constraints for AUTOSAR module configuration parameters. It acts as the schema that tools use to validate and generate ARXML configuration.

XDM Core Concepts

  • Containers — Grouping elements that hold parameters (e.g., ComSignal, CanIfChannel)
  • Parameters — Typed values: integers, floats, booleans, enumerations, strings
  • References — Links between containers (e.g., a signal referencing a PDU)
  • Multiplicity — Cardinality constraints (e.g., exactly one EcuMConfig, 0..* ComIPdu)

XDM-Based Workflow

  1. Define the XDM schema for each BSW module (parameter definition)
  2. Use the schema to instantiate configurations in ARXML
  3. Validate the ARXML against XDM constraints
  4. Generate C code from the validated configuration
Note: XDM is defined by the AUTOSAR standard in the BSW Module Description. Vendor tools (e.g., EB tresos, Vector DaVinci) use XDM to provide guided configuration editors.

RTE Mapping

The RTE (Runtime Environment) is a generated layer that provides the communication infrastructure between SWCs and between SWCs and BSW modules. RTE mapping is the process of connecting ports, mapping runnables to tasks, and defining the data flow.

RTE Mapping Steps

  1. Port Connection — Connect PPort to RPort between SWCs in the composition
  2. Data Mapping — Map S/R data elements to specific signal paths
  3. Task Mapping — Assign each runnable to an OS task with appropriate scheduling
  4. Exclusive Area Mapping — Define concurrency protection strategy (interrupt lock, task lock, OS resource)
  5. RTE Generation — Generate the Rte.c / Rte.h files from the complete mapping

Common Mistake: Forgetting to map exclusive areas for runnables sharing data across different tasks. This leads to race conditions that are hard to debug on target hardware.
