Vue d'ensembleOverview

Le module FiM (Function Inhibition Manager) agit comme un gardien entre le Dem et les fonctions applicatives. À chaque cycle, le Dem notifie le FiM du statut de chaque événement diagnostique. Le FiM calcule alors une permission booléenne pour chaque Function Identifier (FID) selon les conditions d'inhibition configurées. Un SWC appelle FiM_GetFunctionPermission() avant d'exécuter une fonction critique — si la permission est FALSE, il adopte un comportement de repli.The FiM (Function Inhibition Manager) acts as a gatekeeper between the Dem and application functions. Each cycle, the Dem notifies FiM of each diagnostic event status. FiM then computes a boolean permission for each Function Identifier (FID) based on configured inhibition conditions. A SWC calls FiM_GetFunctionPermission() before executing a critical function — if the permission is FALSE, it falls back to a safe default.

Déclaration d'un FID lié à un événement Dem en ARXML, et interrogation de la permission dans un runnable C.FID declaration linked to a Dem event in ARXML, and permission query inside a C runnable.

ARXML 
<FIM-FID>
  <SHORT-NAME>
    FiMFid_MotorControl
  </SHORT-NAME>

  <FIM-INHIBITION-CONFIGURATION>
    <FIM-INHIBITING-EVENT-REF>
      /Dem/DemEventParameter/
      SensorOverTemp
    </FIM-INHIBITING-EVENT-REF>

    <FIM-INHIBITION-MASK>
      LAST_FAILED
    </FIM-INHIBITION-MASK>
  </FIM-INHIBITION-CONFIGURATION>

</FIM-FID>
C — SWC Runnable 
#include "FiM.h"

/* Generated FID constant */
#define FiMConf_FiMFID_MotorControl  0U

void MotorControl_Run(void)
{
  boolean permission = FALSE;

  /* Query permission before
   * executing critical function */
  (void)FiM_GetFunctionPermission(
    FiMConf_FiMFID_MotorControl,
    &permission
  );

  if (permission == TRUE)
  {
    /* Normal operation */
    Motor_SetTorque(
      MotorControl_CalcTorque()
    );
  }
  else
  {
    /* Safe fallback — DTC active */
    Motor_SetTorque(0);
  }
}

Bonnes pratiquesBest Practices

Conseil : Choisissez le masque d'inhibition avec soin — LAST_FAILED inhibe dès le premier échec non qualifié, TESTED_AND_FAILED attend la qualification complète par le Dem. Pour les fonctions de sécurité critique, préférez NOT_TESTED pour bloquer également lors des phases de démarrage avant que le monitoring ne soit exécuté.Tip: Choose the inhibition mask carefully — LAST_FAILED inhibits on the first unqualified failure, TESTED_AND_FAILED waits for full Dem qualification. For safety-critical functions, prefer NOT_TESTED to also block during startup before monitoring has run.
