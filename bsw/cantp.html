Vue d'ensembleOverview

Le CanTp implémente le protocole ISO 15765-2 pour transporter les messages multi-trames sur CAN. Il est principalement utilisé pour les communications diagnostic (UDS via Dcm) mais aussi pour les grands PDUs applicatifs. Il gère les types de trames SF, FF, CF et FC avec les paramètres BS (Block Size), STmin et les timeouts N_As/N_Bs/N_Cs.CanTp implements the ISO 15765-2 protocol for multi-frame message transport on CAN. It is primarily used for diagnostic communications (UDS via Dcm) but also for large application PDUs. It manages frame types SF, FF, CF and FC with BS (Block Size), STmin parameters and N_As/N_Bs/N_Cs timeouts.

C — CanTp Segmentation 
/* Multi-frame RX sequence:
 *
 * Tester → ECU:
 *
 * [FF] First Frame
 *  PCI: 0x10 | length
 *  Payload: first 6 bytes
 *
 * ECU → Tester:
 * [FC] Flow Control
 *  BS=0, STmin=10ms
 *  (accept all CFs, 10ms apart)
 *
 * Tester → ECU:
 * [CF] Consecutive Frame (SN=1)
 *  7 bytes payload
 * [CF] SN=2, SN=3...
 *  Until complete message received
 */

/* CanTp_MainFunction() manages:
 * - FC transmission timing
 * - CF sequence number check
 * - N_Cr timeout monitoring
 * - Buffer management via
 *   PduR_CanTpCopyRxData()
 *
 * On completion:
 *   PduR_CanTpRxIndication()
 *   → Dcm processes UDS request
 */

Bonnes pratiquesBest Practices

Conseil : Calibrez le STmin en fonction de la charge bus — un STmin trop faible provoque des pertes de trames. Activez le padding pour éviter les trames de taille variable qui compliquent le debug. Les timeouts N_Ar/N_Br/N_Cr doivent être conformes à la spécification OEM.Tip: Calibrate STmin based on bus load — a too-low STmin causes frame loss. Enable padding to avoid variable-length frames that complicate debugging. N_Ar/N_Br/N_Cr timeouts must comply with OEM specification.
