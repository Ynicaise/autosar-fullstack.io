Développement CDDCDD Development

Un Complex Device Driver (CDD) fournit une interface SWC standardisée tout en ayant un accès direct aux registres hardware du microcontrôleur sans passer par la couche d'abstraction MCAL.A Complex Device Driver (CDD) provides a standardized SWC interface while having direct access to microcontroller hardware registers without going through the MCAL abstraction layer.

Quand utiliser un CDDWhen to Use a CDD

  • Périphériques hardware non couverts par les drivers MCAL standardsHardware peripherals not covered by standard MCAL drivers
  • Opérations critiques en temps nécessitant un accès direct aux registresTime-critical operations requiring direct register access
  • Protocoles de communication spécifiques non gérés par les piles standardsCustom communication protocols not handled by standard stacks
  • Intégration de drivers legacy dans une architecture AUTOSARLegacy driver integration into an AUTOSAR architecture

Exemples CDD — ARXML & Code CCDD Examples — ARXML & C Code

Un CDD est déclaré comme un SWC standard en ARXML mais son implémentation C accède directement aux registres hardware.A CDD is declared as a standard SWC in ARXML but its C implementation directly accesses hardware registers.

CDD Déclaration CDD avec accès registre directCDD Declaration with Direct Register Access

Le type SWC CDD est déclaré en ARXML comme tout autre composant, mais le code C implémente un accès direct aux registres du périphérique (ici un capteur SPI custom).The CDD SWC type is declared in ARXML like any other component, but the C code implements direct access to the peripheral registers (here a custom SPI sensor).

C — CDD_CustomSensor.c 
#include "Rte_CDD_CustomSensor.h"

/* Hardware register addresses */
#define SPI2_BASE   0x40003800UL
#define SPI_DR      (*(volatile uint16*)\
                     (SPI2_BASE + 0x0CU))
#define SPI_SR      (*(volatile uint16*)\
                     (SPI2_BASE + 0x08U))
#define SPI_SR_RXNE (1U << 0U)
#define SPI_SR_TXE  (1U << 1U)

/* CDD Runnable — direct register I/O */
VOID CDD_ReadSensor(VOID)
{
  uint16 rawData;

  /* Wait for TX buffer empty */
  while (!(SPI_SR & SPI_SR_TXE)) {}

  /* Send read command */
  SPI_DR = 0x8000U; /* Read cmd */

  /* Wait for data reception */
  while (!(SPI_SR & SPI_SR_RXNE)) {}

  rawData = SPI_DR;

  /* Convert & provide via RTE port */
  float32 voltage =
    (float32)rawData * 3.3f / 4096.0f;

  (void)Rte_Write_PPort_SensorData(
    voltage
  );
}

C — CDD_CustomSensor.c 
#define RCC_APB1ENR (*(volatile uint32*)\
                     (0x40023840UL))
#define SPI2_CR1    (*(volatile uint16*)\
                     (SPI2_BASE + 0x00U))
#define SPI2_CR2    (*(volatile uint16*)\
                     (SPI2_BASE + 0x04U))
#define NVIC_ISER1  (*(volatile uint32*)\
                     (0xE000E104UL))

VOID CDD_Sensor_Init(VOID)
{
  /* 1. Enable SPI2 peripheral clock */
  RCC_APB1ENR |= (1U << 14U);

  /* 2. Configure SPI: Master, 8-bit,
   *    CPOL=0, CPHA=0, /16 prescaler */
  SPI2_CR1 = (1U << 2U)   /* MSTR */
           | (3U << 3U);  /* BR /16 */

  /* 3. Enable RX interrupt */
  SPI2_CR2 = (1U << 6U); /* RXNEIE */

  /* 4. Enable NVIC IRQ #36 (SPI2) */
  NVIC_ISER1 = (1U << 4U);

  /* 5. Enable SPI */
  SPI2_CR1 |= (1U << 6U); /* SPE */
}
CDD vs MCAL : Contrairement aux drivers MCAL qui exposent des APIs standardisées (Spi_AsyncTransmit), un CDD manipule directement les registres. L'interface AUTOSAR est préservée côté port (S/R, C/S) mais l'implémentation est spécifique au hardware.CDD vs MCAL: Unlike MCAL drivers that expose standardized APIs (Spi_AsyncTransmit), a CDD directly manipulates registers. The AUTOSAR interface is preserved on the port side (S/R, C/S) but the implementation is hardware-specific.

Conception des RunnablesRunnable Design

Une Runnable Entity est la plus petite unité de code ordonnançable dans un SWC AUTOSAR. Chaque runnable est déclenchée par un événement RTE et mappée sur une tâche OS.A Runnable Entity is the smallest schedulable unit of code in an AUTOSAR SWC. Each runnable is triggered by an RTE Event and mapped to an OS Task.

Mapping Runnable-to-TaskRunnable-to-Task Mapping

Chaque runnable doit être mappée sur une tâche OS. La stratégie de mapping affecte le comportement temps réel :Each runnable must be mapped to an OS task. The mapping strategy affects real-time behavior:

  • Une runnable par tâche — Isolation maximale, overhead le plus élevéOne runnable per task — Maximum isolation, highest overhead
  • Plusieurs runnables par tâche — Contexte de tâche partagé, exécution séquentielle au sein de la tâcheMultiple runnables per task — Shared task context, sequential execution within task
  • Runnables Catégorie 1 vs Catégorie 2 — Les runnables Cat1 peuvent préempter les Cat2 et vice versa selon la priorité de tâcheCategory 1 vs Category 2 runnables — Cat1 runnables can preempt Cat2 and vice versa depending on task priority
Conseil : Regroupez les runnables de même période dans la même tâche pour minimiser l'overhead de changement de contexte tout en maintenant un ordonnancement déterministe.Tip: Group runnables with the same period into the same task to minimize context switch overhead while maintaining deterministic scheduling.
