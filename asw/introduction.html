Vue d'ensemble de la couche ASWASW Layer Overview

La couche Application Software (ASW) est le niveau le plus haut de l'architecture AUTOSAR Classic. Elle contient toute la logique métier de l'ECU sous forme de Software Components (SWC) qui communiquent exclusivement via le RTE (Runtime Environment).The Application Software (ASW) layer is the highest level of the AUTOSAR Classic architecture. It contains all the ECU business logic in the form of Software Components (SWC) that communicate exclusively through the RTE (Runtime Environment).

Chaque SWC est indépendant du hardware : il ne connaît ni le microcontrôleur cible ni les modules BSW sous-jacents. Cette séparation garantit la portabilité et la réutilisabilité des composants applicatifs entre différents projets ECU.Each SWC is hardware-independent: it knows neither the target microcontroller nor the underlying BSW modules. This separation guarantees the portability and reusability of application components across different ECU projects.

Ports & InterfacesPorts & Interfaces

Chaque SWC communique via des ports typés :Each SWC communicates through typed ports:

  • Sender-Receiver (S/R) — Échange de données asynchrone avec accès implicite ou expliciteSender-Receiver (S/R) — Asynchronous data exchange with implicit or explicit access
  • Client-Server (C/S) — Pattern d'appel de fonction synchrone ou asynchroneClient-Server (C/S) — Synchronous or asynchronous function call pattern
  • Mode-Switch — Notifications de mode (ex. : changements de mode EcuM)Mode-Switch — Mode notifications (e.g., EcuM mode changes)
  • Trigger — Signaux de déclenchement événementielTrigger — Event-driven trigger signals
  • NvData — Interface de données non-volatiles pour accès NvMNvData — Non-volatile data interface for NvM access

Runnables & ÉvénementsRunnables & Events

Une Runnable Entity est la plus petite unité de code ordonnançable dans un SWC. Chaque runnable est déclenchée par un événement RTE et mappée sur une tâche OS.A Runnable Entity is the smallest schedulable unit of code in a SWC. Each runnable is triggered by an RTE Event and mapped to an OS Task.

Exemples détaillés par type d'événementDetailed Examples by Event Type

Chaque type d'événement RTE possède une description ARXML distincte et génère un squelette C spécifique. Les exemples ci-dessous illustrent les 5 patterns fondamentaux.Each RTE event type has a distinct ARXML description and generates a specific C skeleton. The examples below illustrate the 5 fundamental patterns.

TimingEvent Déclenchement périodique — boucle de contrôle 10 msPeriodic trigger — 10 ms control loop

Le TimingEvent lie une RunnableEntity à une période fixe (en secondes). Le RTE génère l'appel depuis la tâche OS correspondante. C'est l'événement le plus courant dans les SWC de contrôle.The TimingEvent binds a RunnableEntity to a fixed period (in seconds). The RTE generates the call from the corresponding OS task. This is the most common event in control SWCs.

ARXML 
<SWC-INTERNAL-BEHAVIOR>
  <SHORT-NAME>EngineCtrl_IB</SHORT-NAME>
  <RUNNABLES>
    <RUNNABLE-ENTITY>
      <SHORT-NAME>
        RE_EngineCtrl_10ms
      </SHORT-NAME>
      <CAN-BE-INVOKED-CONCURRENTLY>
        false
      </CAN-BE-INVOKED-CONCURRENTLY>
    </RUNNABLE-ENTITY>
  </RUNNABLES>
  <EVENTS>
    <TIMING-EVENT>
      <SHORT-NAME>
        TE_EngineCtrl_10ms
      </SHORT-NAME>
      <START-ON-EVENT-REF
        DEST="RUNNABLE-ENTITY">
        /EngineCtrl/RE_EngineCtrl_10ms
      </START-ON-EVENT-REF>
      <PERIOD>0.01</PERIOD>
    </TIMING-EVENT>
  </EVENTS>
</SWC-INTERNAL-BEHAVIOR>
C — EngineCtrl_Runnable.c 
/* Runnable : RE_EngineCtrl_10ms
 * Trigger  : TimingEvent — 10 ms
 * OS Task  : OsTask_10ms         */
FUNC(void, ENGINECTRL_CODE)
RE_EngineCtrl_10ms(void)
{
  float32 speed;
  float32 torqueCmd;

  (void)Rte_Read_RPort_Speed_Speed(
    &speed);

  torqueCmd =
    EngineCtrl_ComputeTorque(speed);

  (void)Rte_Write_PPort_TorqueCmd(
    torqueCmd);
}
InitEvent Initialisation du SWC au démarrage du RTESWC initialization at RTE startup

L'InitEvent déclenche la runnable d'initialisation une seule fois, lors de l'appel de Rte_Start(), avant l'entrée dans les tâches cycliques. Aucun attribut PERIOD : l'événement est one-shot.The InitEvent triggers the init runnable exactly once during Rte_Start(), before cyclic tasks begin. No PERIOD attribute: the event is one-shot.

ARXML 
<SWC-INTERNAL-BEHAVIOR>
  <SHORT-NAME>EngineCtrl_IB</SHORT-NAME>
  <RUNNABLES>
    <RUNNABLE-ENTITY>
      <SHORT-NAME>
        RE_EngineCtrl_Init
      </SHORT-NAME>
      <CAN-BE-INVOKED-CONCURRENTLY>
        false
      </CAN-BE-INVOKED-CONCURRENTLY>
    </RUNNABLE-ENTITY>
  </RUNNABLES>
  <EVENTS>
    <INIT-EVENT>
      <SHORT-NAME>
        IE_EngineCtrl_Init
      </SHORT-NAME>
      <START-ON-EVENT-REF
        DEST="RUNNABLE-ENTITY">
        /EngineCtrl/RE_EngineCtrl_Init
      </START-ON-EVENT-REF>
    </INIT-EVENT>
  </EVENTS>
</SWC-INTERNAL-BEHAVIOR>
C — EngineCtrl_Runnable.c 
/* Runnable : RE_EngineCtrl_Init
 * Trigger  : InitEvent — one-shot
 * Called by: Rte_Start()          */
FUNC(void, ENGINECTRL_CODE)
RE_EngineCtrl_Init(void)
{
  EngineCtrl_State.torque    = 0.0f;
  EngineCtrl_State.mode      =
    CTRL_MODE_OFF;
  EngineCtrl_State.errorFlag = FALSE;

  (void)Rte_Write_PPort_TorqueCmd(
    0.0f);
}
DataReceivedEvent Traitement événementiel d'une nouvelle donnée S/REvent-driven processing of new S/R data

Le DataReceivedEvent déclenche la runnable dès qu'une nouvelle valeur est disponible sur le R-Port S/R ciblé. La référence DATA-IREF identifie précisément le port et l'élément de données. Utiliser Rte_Receive (accès explicite) pour lire la valeur reçue.The DataReceivedEvent triggers the runnable as soon as a new value is available on the targeted S/R R-Port. The DATA-IREF reference precisely identifies the port and data element. Use Rte_Receive (explicit access) to read the received value.

ARXML 
<DATA-RECEIVED-EVENT>
  <SHORT-NAME>DRE_OnSpeed</SHORT-NAME>
  <START-ON-EVENT-REF
    DEST="RUNNABLE-ENTITY">
    /EngineCtrl/RE_OnSpeedReceived
  </START-ON-EVENT-REF>
  <DATA-IREF>
    <PORT-PROTOTYPE-REF
      DEST="R-PORT-PROTOTYPE">
      /EngineCtrl/RPort_Speed
    </PORT-PROTOTYPE-REF>
    <TARGET-DATA-PROTOTYPE-REF
      DEST=
      "VARIABLE-DATA-PROTOTYPE">
      /Interfaces/If_SpeedData/Speed
    </TARGET-DATA-PROTOTYPE-REF>
  </DATA-IREF>
</DATA-RECEIVED-EVENT>
C — EngineCtrl_Runnable.c 
/* Runnable : RE_OnSpeedReceived
 * Trigger  : DataReceivedEvent
 *            RPort_Speed / Speed  */
FUNC(void, ENGINECTRL_CODE)
RE_OnSpeedReceived(void)
{
  float32 speed;
  Std_ReturnType ret;

  ret = Rte_Receive_RPort_Speed_Speed(
    &speed);

  if (ret == RTE_E_OK)
  {
    EngineCtrl_State.lastSpeed = speed;
    EngineCtrl_ProcessSpeed(speed);
  }
}
ModeSwitchEvent Réaction aux transitions de mode (EcuM RUN)Reacting to mode transitions (EcuM RUN)

Le SWC-MODE-SWITCH-EVENT déclenche une runnable lors de l'entrée (ENTERED) ou de la sortie (EXITED) d'un mode déclaré dans un ModeDeclarationGroup. Il faut une runnable et un événement distincts par direction de transition.The SWC-MODE-SWITCH-EVENT triggers a runnable on entry (ENTERED) or exit (EXITED) from a mode declared in a ModeDeclarationGroup. A separate runnable and event are required for each transition direction.

ARXML 
<SWC-MODE-SWITCH-EVENT>
  <SHORT-NAME>MSE_Run_Enter</SHORT-NAME>
  <START-ON-EVENT-REF
    DEST="RUNNABLE-ENTITY">
    /EngineCtrl/RE_OnRun_Enter
  </START-ON-EVENT-REF>
  <ACTIVATION>ENTERED</ACTIVATION>
  <MODE-IREF>
    <REQUIRING-PORT-PROTOTYPE-REF
      DEST="R-PORT-PROTOTYPE">
      /EngineCtrl/RPort_EcuMMode
    </REQUIRING-PORT-PROTOTYPE-REF>
    <TARGET-MODE-DECLARATION-REF
      DEST="MODE-DECLARATION">
      /AUTOSAR/EcuM/Modes/RUN
    </TARGET-MODE-DECLARATION-REF>
  </MODE-IREF>
</SWC-MODE-SWITCH-EVENT>

<SWC-MODE-SWITCH-EVENT>
  <SHORT-NAME>MSE_Run_Exit</SHORT-NAME>
  <START-ON-EVENT-REF
    DEST="RUNNABLE-ENTITY">
    /EngineCtrl/RE_OnRun_Exit
  </START-ON-EVENT-REF>
  <ACTIVATION>EXITED</ACTIVATION>
  <!-- same MODE-IREF -->
</SWC-MODE-SWITCH-EVENT>
C — EngineCtrl_Runnable.c 
/* Trigger : ModeSwitchEvent
 *           ENTERED EcuM/RUN      */
FUNC(void, ENGINECTRL_CODE)
RE_OnRun_Enter(void)
{
  EngineCtrl_State.mode =
    CTRL_MODE_RUN;
  EngineCtrl_EnableActuators();
}

/* Trigger : ModeSwitchEvent
 *           EXITED EcuM/RUN       */
FUNC(void, ENGINECTRL_CODE)
RE_OnRun_Exit(void)
{
  EngineCtrl_State.mode =
    CTRL_MODE_OFF;
  EngineCtrl_DisableActuators();
}
OperationInvokedEvent Handler côté serveur d'un port Client-ServerServer-side handler for a Client-Server port

L'OperationInvokedEvent déclenche la runnable serveur lorsque le client appelle l'opération via le RTE. La runnable implémente le corps de la fonction serveur et retourne un Std_ReturnType. Le OPERATION-IREF cible le P-Port et l'opération exposée.The OperationInvokedEvent triggers the server runnable when the client invokes the operation through the RTE. The runnable implements the server function body and returns a Std_ReturnType. The OPERATION-IREF targets the P-Port and the exposed operation.

ARXML 
<OPERATION-INVOKED-EVENT>
  <SHORT-NAME>
    OIE_GetFaultCount
  </SHORT-NAME>
  <START-ON-EVENT-REF
    DEST="RUNNABLE-ENTITY">
    /DiagSrv/RE_GetFaultCount
  </START-ON-EVENT-REF>
  <OPERATION-IREF>
    <CONTEXT-P-PORT-REF
      DEST="P-PORT-PROTOTYPE">
      /DiagSrv/PPort_DiagService
    </CONTEXT-P-PORT-REF>
    <TARGET-PROVIDED-OPERATION-REF
      DEST=
      "CLIENT-SERVER-OPERATION">
      /Interfaces/If_DiagService
        /GetFaultCount
    </TARGET-PROVIDED-OPERATION-REF>
  </OPERATION-IREF>
</OPERATION-INVOKED-EVENT>
C — DiagSrv_Runnable.c 
/* Runnable : RE_GetFaultCount
 * Trigger  : OperationInvokedEvent
 * Server   : PPort_DiagService
 *            / GetFaultCount       */
FUNC(Std_ReturnType, DIAGSRV_CODE)
RE_GetFaultCount(
  P2VAR(uint16, AUTOMATIC,
    RTE_APPL_DATA) FaultCount)
{
  *FaultCount =
    DiagSrv_CountActiveFaults();

  return RTE_E_OK;
}
